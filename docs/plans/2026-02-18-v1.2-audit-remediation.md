## Phase 1: CRITICAL Fixes (Blocks Correct Operation)

### 1A. Fix `exit_plan_mode` Path Directory (O-001, S-002)

**Files:**
- `GEMINI.md` (line 51)
- `skills/implementation-planning/SKILL.md` (lines 169-171, 212)

**Problem:** `exit_plan_mode` is called with project-local `.gemini/plans/...` paths, but the tool validates that `plan_path` is within `~/.gemini/tmp/<project>/<session-id>/plans/`. Plan Mode also restricts `write_file` to that tmp directory, so the "access error fallback" workaround cannot work.

**Assertion:** `exit_plan_mode` validates plan_path is within `storage.getProjectTempPlansDir()` via `isSubpath()`. Plan Mode restricts `write_file` to the tmp plans directory only.

**Remediation:**
1. In `skills/implementation-planning/SKILL.md`, update the Phase 2 approval protocol:
   - Write the implementation plan to `~/.gemini/tmp/plans/` during Plan Mode (the only writable location)
   - Call `exit_plan_mode` with the tmp-directory path
   - After user approval, copy to `.gemini/plans/YYYY-MM-DD-<slug>-impl-plan.md`
2. In `GEMINI.md`, update Phase 2 transition to match the two-step process
3. Remove all "if the tool returns an access error" fallback language

### 1B. Replace Fabricated `delegate_to_agent` References (O-002)

**Files:**
- `docs/architecture/system-overview.md` (lines 173, 365, 470, 505, 555, 707)
- `docs/architecture/agent-system.md` (lines 86, 224, 434, 452-461, 505, 782)

**Problem:** Architecture docs describe a non-existent `delegate_to_agent(agent: "name", prompt: "...")` tool. The actual mechanism is each agent in `agents/` is auto-registered as a callable tool matching its `name` field.

**Assertion:** Gemini CLI exposes each subagent as a tool of the same name, per `docs/core/subagents.md`: "Sub-agents are exposed to the main agent as a tool of the same name."

**Remediation:**
1. Replace all `delegate_to_agent` references with correct mechanism: "Call the `<agent-name>` tool"
2. Update examples from `delegate_to_agent(agent: "coder", prompt: "...")` to `coder(prompt: "...")`
3. Add note: agents in `agents/` are auto-registered as tools at extension load time

## Phase 2: HIGH Fixes (Incorrect Behavior)

### 2A. Fix `write_todos` Exclusivity Violation (S-001, O-008)

**Files:**
- `skills/execution/SKILL.md` (lines 47, 62-63)

**Problem:** Parallel batch execution marks multiple todos `in_progress` simultaneously, violating: "Only one task can be marked `in_progress` at any time."

**Assertion:** `docs/tools/todos.md` states the exclusivity constraint.

**Remediation:**
1. For parallel batches, create a single composite todo: "Executing batch: Phase N, M, O"
2. Mark only that single composite todo as `in_progress`
3. Track individual phase status in session state YAML (not write_todos)

### 2B. Make Plan Mode Conditional on Experimental Flag (O-003)

**Files:**
- `GEMINI.md` (lines 43-44)
- `skills/design-dialogue/SKILL.md`

**Problem:** GEMINI.md unconditionally calls `enter_plan_mode`, but Plan Mode is experimental requiring `experimental.plan: true`.

**Assertion:** Plan Mode requires opt-in per `docs/cli/plan-mode.md`: "To use Plan Mode, enable it via `/settings` or add `experimental.plan: true`."

**Remediation:**
1. Make `enter_plan_mode` conditional: "If Plan Mode is available, call it; otherwise proceed in normal mode with `ask_user` approval gates"
2. Add fallback behavior in design-dialogue skill

### 2D. Fix Architecture Docs: YOLO Mode vs Agent Loading (O-004)

**Files:**
- `docs/architecture/agent-system.md` (lines 224-228, 505-511)

**Problem:** Docs say "headless processes don't load agent definitions." Wrong — agent definitions ARE loaded, and agent `tools:` restrictions are enforced via tool registry filtering (not policy). YOLO bypasses policy approval but NOT registry filtering.

**Assertion [VALIDATED]:** Agent `tools:` frontmatter is enforced via `ToolRegistry` in `local-executor.ts` — only listed tools are registered, making unlisted tools invisible to the model. This is NOT a policy check, so YOLO cannot bypass it. YOLO only bypasses the policy engine's confirmation requirement for tools the agent already has access to.

**Remediation:**
1. Correct explanation: agent tool restrictions are enforced at the registry level (unlisted tools are not registered), which YOLO cannot bypass
2. Agent definitions ARE loaded when extension is linked
3. Prompt-based TOOL RESTRICTIONS are defense-in-depth for the case where `--yolo` auto-approves confirmations on tools the agent DOES have, not because tools aren't restricted
4. Clarify that `--yolo` bypasses confirmation dialogs, not tool availability

### 2E. ~~Verify `read_file` Behavior for `.gemini/` Paths~~ ORIGINAL ASSERTION WRONG (O-005)

**Files:**
- `skills/session-management/SKILL.md` (lines 29-48)

**Remediation:**

2. Update the skills to clearly explain WHY the workaround is needed: "`read_file` enforces `.gitignore` patterns, and `.gemini/` is typically gitignored"
3. Ensure `.gemini/` IS in the project's `.gitignore` (this is the assumed pattern)
4. The `write_file` asymmetry (S-005) should also be investigated — check if `write_file` has the same ignore check
